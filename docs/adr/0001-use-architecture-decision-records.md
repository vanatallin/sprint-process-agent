# 1. Use Architecture Decision Records

Date: 2026-03-25

## Status

Accepted

## Context

We need to document architectural decisions in this project so that:
- Future contributors understand why decisions were made
- We can revisit decisions when context changes
- Knowledge isn't lost when team members change

## Decision

We will use Architecture Decision Records (ADRs) as described by Michael Nygard.

ADRs will:
- Be stored in `docs/adr/`
- Be numbered sequentially (0001, 0002, etc.)
- Use Markdown format
- Include Status, Context, Decision, and Consequences sections

## Consequences

- Decisions are documented with context
- Future contributors understand rationale
- ADRs are lightweight and version-controlled
- Requires discipline to create ADRs for significant decisions
